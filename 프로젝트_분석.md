# 프로젝트 분석

## 1. 프로젝트 개요

### 1.1 프로젝트 소개
Django 기반의 개인 블로그 및 포트폴리오 웹 애플리케이션이다.

[Do it! 장고+부트스트랩 파이썬 웹 개발의 정석]을 기반으로 개발을 시작하였으며, 최신 개발 환경에 맞게 수정 및 개선하였다. 현재 최종적으로 Docker 기반 운영 환경을 구성하여 AWS Lightsail 서버에 실제 서비스를 배포하였다.

### 1.2 프로젝트 목표

- Django 웹 개발 학습
- 프로젝트를 처음부터 끝까지 완성하는 경험
- 실제 운영 가능한 웹 서비스의 구축 경험
- Docker 기반 배포 환경 구성 경험
- AWS 및 DNS 서비스 연결 경험
- 최신 개발 환경에 적응하는 경험

### 1.3 개발 기간

2025.8.7~2026.7.7(11개월)

실제 작업 일수 : 68일(약 2개월 8일)

- Django 기반 웹 애플리케이션 구현

## 2. 프로젝트 진행 과정

### 2.1 프로젝트 시작과 방향 변경

프로젝트는 처음부터 교재를 그대로 따라가는 방식으로 시작하지 않았다.

초기에는 교재의 내용을 참고하여 `maps`라는 앱을 중심으로 독자적인 프로젝트를 진행하고자 하였다. 그러나 Django에 대한 이해가 충분하지 않은 상태에서 교재를 실시간으로 참고하며 필요한 기능만 선별하여 구현하는 것은 난이도가 높았고, 프로젝트를 안정적으로 진행하기 어렵다고 판단하였다.

이에 기존 프로젝트를 정리한 후, 교재의 흐름을 먼저 완성하는 방향으로 계획을 변경하였다. 이후 `blog`와 `single_pages` 애플리케이션을 새롭게 구성하여 프로젝트를 처음부터 다시 진행하였다.

이러한 방향 전환을 통해 Django의 구조와 개발 흐름을 먼저 익힌 뒤, 이후 프로젝트를 확장하는 기반을 마련하고자 하였다.

---

### 2.2 개발 방식

프로젝트는 교재를 기반으로 진행하였으나, 교재의 내용을 그대로 복사하거나 적용하는 방식으로 개발하지는 않았다.

교재에서 제시한 기능과 구현 방법을 따라가면서도 현재 개발 환경에서 발생하는 문제를 별도로 해결하였다.

문제가 발생하면 다음과 같은 과정을 반복하였다.

1. 오류 내용을 확인하고 원인을 추정한다.
2. AI에게 현재 오류가 버전 차이 또는 환경 차이로 인해 발생한 것인지 확인한다.
3. AI의 설명을 그대로 적용하지 않고 Django, Bootstrap, Docker, 기타 라이브러리 등의 공식 문서를 확인한다.
4. 프로젝트에 맞도록 수정한 뒤 동작을 검증한다.

또한 디자인이나 사용자 경험에 대해서는 교재를 그대로 따르기보다 프로젝트의 목적 및 자체적 판단에 따라 필요한 부분을 직접 수정하였다.

---

### 2.3 개발 기록

프로젝트의 진행 과정은 개발일지를 중심으로 기록하였다.

초기에는 매일 수행한 작업과 느낀 점을 중심으로 작성하였으며, 프로젝트가 진행될수록 작성 형식을 점차 개선하였다.

개발일지에는 다음과 같은 내용을 지속적으로 기록하였다.

- 작업한 기능
- 발생한 문제
- 해결 과정
- 느낀 점
- 향후 개선 사항

프로젝트 중반부터는 개발일지와 별도로 TIL(Today I Learned) 문서를 작성하였다.

TIL에는 프로젝트를 진행하면서 새롭게 접한 기술이나 라이브러리, 용어 등을 간략하게 조사하여 정리하였다. 개발일지가 프로젝트의 진행 과정을 기록하는 문서라면, TIL은 프로젝트를 통해 학습한 내용을 정리하는 문서로 구분하여 관리하였다. 해당 사항은 프로젝트 중 학습한 내용 외 별도 학습 사항을 포괄하여 별도의 repository로 관리되고 있다.



## 3. 교재와 내 역할

프로젝트는 『장고+부트스트랩 파이썬 웹 개발의 정석』을 기반으로 진행하였다.

교재를 따라 Django의 기본 구조와 기능을 구현하였으며, 프로젝트가 진행될수록 현재 개발 환경과의 차이, 외부 서비스 변화, 프로젝트 목적에 따른 요구사항이 발생하였다.

이러한 부분은 교재의 내용을 그대로 적용하지 않고 직접 조사하고 수정하여 프로젝트를 완성하였다.

---

### 3.1 교재에서 제공한 내용

교재를 통해 다음과 같은 내용을 구현하였다.

#### 프로젝트 기본 구조

- Django 프로젝트 및 애플리케이션 구성
- Model, View, URL, Template 구조
- Bootstrap 기반 화면 구성

#### 블로그 기능

- 게시글 작성, 수정, 삭제
- 댓글 기능
- 카테고리
- 태그
- 검색 기능
- Markdown 작성 기능 및 실시간 프리뷰

#### 부가 기능

- Google 로그인
- Landing Page
- Portfolio
- About Me

#### 배포

- Docker
- Gunicorn
- Nginx
- PostgreSQL
- AWS Lightsail 배포
- HTTPS 적용

※ 위 기능은 교재의 흐름을 따라 구현하였다.

---

### 3.2 직접 수행한 내용

교재의 내용을 그대로 적용하기 어려운 부분은 직접 조사하고 프로젝트에 맞게 수정하였다.

#### 최신 개발 환경 대응

교재 집필 당시와 현재의 개발 환경 차이로 인해 발생한 문제를 최신 환경에 맞게 수정하였다.

- Bootstrap 5에 맞게 클래스 및 속성 수정
- Docker CLI 플러그인 통합에 따른 명령어 변경(Compose V1 -> V2) 적용
- 최신 Django 환경에서 발생한 버전 차이 수정

---

#### 프로젝트에 맞는 UI 및 사용성 개선

교재의 기본 디자인을 그대로 사용하지 않고 프로젝트 목적과 사용성을 고려하여 일부 화면을 수정하였다.

- 카드 디자인 및 레이아웃 수정
- Navigation Bar 디자인 변경 및 관련 레이아웃 수정
- Portfolio 이미지 비율에 따른 배치 개선
- 페이지네이션을 '이전/다음' 중심에서 '맨 앞/숫자/맨 뒤' 구조로 개선하여 원하는 페이지로 빠르게 이동할 수 있도록 수정


---

### 교재 외 문제 해결

교재에서 다루지 않았거나 현재 환경에서 발생한 문제를 직접 조사하고 해결하였다.

- Markdown 출력 구조 개선 및 보안 고려 렌더링 구현
- 검색어의 특수 문자(/) 포함으로 발생하는 URL 처리 문제 개선
- 교재에서 사용한 Avatar 서비스 종료에 대응하여 UI Avatars로 변경
- Google OAuth를 HTTPS 환경에 맞게 재설정
- AWS Lightsail의 메모리 부족 문제를 Swap 구성으로 해결

---

### 개발 과정 개선

프로젝트 진행 과정에서 개발 효율과 유지보수를 위해 다음과 같은 방법을 적용하였다.

- 개발일지 작성 및 작성 방식 개선
- TIL(Today I Learned) 문서 작성
- AI의 설명을 참고한 후 공식 문서를 통해 검증하는 개발 방식 정착
- 향후 개선 사항을 지속적으로 기록하여 기능 확장 후보 관리

## 4. 4. 교재를 벗어나게 된 원인

프로젝트는 교재를 기반으로 진행하였지만, 개발을 진행하면서 교재만으로는 해결할 수 없는 상황이 지속적으로 발생하였다.

이는 단순히 교재가 잘못되었기 때문이 아니라, 교재 집필 이후 개발 환경과 서비스가 변화하고, 프로젝트의 목적에 맞는 추가적인 개선이 필요했기 때문이다.

대표적인 원인은 다음과 같다.

---

### 4.1 프레임워크 및 개발 환경의 변화

교재 집필 이후 Django, Bootstrap 등 주요 프레임워크가 업데이트되면서 일부 코드와 사용 방법이 변경되었다.

또한 Docker Compose의 명령 체계 변경 등 개발 환경도 함께 변화하여 교재의 내용을 그대로 적용할 수 없는 경우가 발생하였다.

이러한 경우에는 공식 문서를 참고하여 현재 개발 환경에 맞게 수정한 후 프로젝트를 진행하였다.

**대표 사례**

- Bootstrap 5 적용
- Docker Compose 명령 변경
- 최신 Django 환경 대응

---

### 4.2 외부 서비스 변화

프로젝트를 진행하는 동안 교재에서 사용한 일부 외부 서비스가 더 이상 제공되지 않거나, 현재의 서비스 정책과 차이가 있는 경우가 있었다.

이러한 변화는 프로젝트 구현 방식에 직접적인 영향을 주었으며, 기존 서비스를 그대로 사용하는 대신 현재 사용할 수 있는 서비스를 조사하고 비교하여 대체 방안을 적용하였다.

또한 인증 서비스의 정책 변화에 따라 기존 설정만으로는 정상적으로 동작하지 않는 부분도 발생하여, 현재 환경에 맞게 설정을 수정하였다.

**대표 사례**

- 교재에서 사용한 Avatar 서비스 종료에 따른 UI Avatars 적용
- Google OAuth의 현재 정책 및 HTTPS 환경에 맞는 설정 적용

---

### 4.3 비용 및 운영 환경 고려

프로젝트를 실제 서비스로 배포하고 지속적으로 운영하는 것을 목표로 하면서, 단순히 교재와 동일한 환경을 구성하기보다 운영 비용과 유지보수성을 함께 고려하였다.

교재에서 제안한 서비스 중 지속적인 비용이 발생하는 항목은 프로젝트 규모와 목적을 고려하여 대체 서비스를 검토하였으며, 무료로 필요한 기능을 제공하는 서비스를 선택하였다.

또한 실제 서버 환경에서는 개발 환경에서 경험하지 못한 운영상의 문제가 발생하였으며, 안정적인 서비스 운영을 위해 이에 대한 해결 방법을 적용하였다.

**대표 사례**

- Docker 공식 문서를 참고한 최신 설치 절차 적용
- AWS Route 53 대신 Cloudflare를 사용하여 도메인 구매 및 DNS 연결

---

### 4.4 프로젝트 목적에 따른 개선

프로젝트는 단순히 교재를 완성하는 것이 아니라 개인 포트폴리오로 활용하는 것을 목표로 하였다.

따라서 교재의 기본 기능을 구현하는 것에서 그치지 않고, 프로젝트의 목적과 사용성을 고려하여 화면 구성과 일부 기능을 개선하였다.

또한 프로젝트를 진행하면서 발견한 개선 사항은 개발일지에 기록하여 이후 기능 확장에 활용할 수 있도록 관리하였다.

**대표 사례**

- 카드 디자인 수정
- Navigation Bar 개선
- Portfolio 이미지 배치 개선
- 페이지네이션 UI 개선
- Markdown 출력 방식 개선

## 6. 미래 개선 사항

개발 진행 기간동안 개발일지에 작성하거나 이후 고안한 향후의 개선 사항들을 정리하였다.

### 6.1 기능 개선

- 댓글 수정 UI 개선
  - 별도의 수정 페이지 대신 현재 페이지에서 바로 수정할 수 있도록 개선

## 7. 프로젝트 성장 기록

프로젝트를 진행하면서 개발 과정과 문제 해결 방식에 변화가 나타났다. 개발일지를 바탕으로 프로젝트를 통해 성장한 내용을 정리하였다.

### 프로젝트 방향 설정

- 교재를 참고하여 독자적인 프로젝트(`maps`)를 진행하려 하였으나, 현재 수준에서는 어려움을 판단하고 프로젝트를 재구성하였다.
- 이후 교재를 기반으로 프로젝트를 완성한 뒤 확장하는 방향으로 개발 계획을 변경하였다.


### 개발 기록의 변화

- 초기에는 작업 내용과 느낀 점을 중심으로 개발일지를 작성하였다.
- 프로젝트가 진행될수록 문제 해결 과정과 원인을 함께 기록하는 방식으로 발전하였다.
- 프로젝트 중반부터는 TIL(Today I Learned)을 작성하여 새롭게 학습한 기술과 개념을 별도로 정리하였다.


### 문제 해결 방식의 변화

- 오류가 발생하면 AI를 통해 원인을 먼저 파악하였다.
- AI의 답변을 그대로 적용하지 않고 공식 문서를 확인하여 내용을 검증하였다.
- 현재 프로젝트에 맞는 해결 방법을 적용하고 개발일지에 기록하였다.



### 기술적 성장

- Django 프로젝트의 전체 구조(Model, View, Template, URL)를 이해하게 되었다.
- Bootstrap을 활용한 화면 구성과 레이아웃 수정 경험을 쌓았다.
- Markdown, OAuth, Docker, Nginx, Gunicorn 등 프로젝트에 필요한 기술을 실제로 적용하였다.
- AWS Lightsail를 이용하여 실제 서비스를 배포하고 운영 환경을 경험하였다.



### 프로젝트 관리

- 개발일지를 지속적으로 작성하며 개발 과정을 기록하였다.
- 프로젝트 진행 중 발견한 개선 사항을 별도로 기록하여 이후 기능 확장에 활용할 수 있도록 관리하였다.
- GitHub를 활용하여 프로젝트의 변경 사항을 단계별로 관리하고, 기능 단위로 개발 이력을 기록하였다.
- 프로젝트 완료 후 개발일지를 기반으로 프로젝트 분석 문서와 README.md를 작성하여 프로젝트를 객관적으로 정리하였다.

## 8. 최종 회고

프로젝트를 시작할 당시 나는 제대로 된 웹 프로젝트를 처음부터 끝까지 완성해 본 경험도, 실제 서비스를 배포해 본 경험도 거의 없었다. 이전에는 주로 예제 코드나 교재의 코드를 따라 작성하며 문법과 구조를 익히는 학습을 진행했으며, 하나의 서비스를 완성한다는 목표보다는 코드를 이해하는 것에 초점을 맞추고 있었다.

이번 프로젝트 역시 처음에는 직접 기능을 설계하여 구현해 보려는 생각으로 시작하였다. 그러나 원하는 기능을 구현하기 위해서는 교재를 부분적으로 참고하는 것만으로는 부족하다는 사실을 깨달았다. Django가 어떤 구조로 동작하는지조차 충분히 이해하지 못한 상태였기 때문에, 결국 처음 계획했던 독자 프로젝트를 중단하고 교재를 기반으로 하나씩 학습하며 프로젝트를 다시 시작하게 되었다.

하지만 프로젝트가 진행될수록 단순히 교재를 따라 작성하는 데서 끝나지 않았다. 교재와 현재 개발 환경의 차이로 인해 다양한 문제가 발생했고, 공식 문서를 참고하여 최신 버전에 맞게 수정하거나 현재 권장되는 방식으로 변경해야 하는 상황을 여러 번 경험했다. 처음에는 교재를 완성하는 것이 목표였다면, 후반부에는 왜 문제가 발생하는지 분석하고 현재 환경에 맞게 수정하는 과정 자체가 프로젝트의 중요한 부분이 되었다.

가장 기억에 남는 경험은 MarkdownX를 적용하는 과정이었다. 단순히 라이브러리를 설치하는 문제가 아니라, Markdown과 HTML이 함께 사용되는 상황에서 예상하지 못한 문제가 발생했다. 게시글에 작성한 HTML 태그가 실제 화면에서 동작하면서 레이아웃을 깨뜨렸고, 교재에서는 다루지 않은 문제였기 때문에 해결 방법을 처음부터 직접 찾아야 했다. AI를 활용하여 여러 방법을 조사하고, 공식 문서를 함께 확인하며 출력 구조를 분석한 끝에 bleach를 이용해 위험한 태그를 제한하는 방식을 적용하였다. 이 과정은 단순히 오류를 해결하는 것을 넘어, 출력 방식과 보안까지 함께 고민했던 경험으로 가장 오래 기억에 남는다.

배포 과정 역시 매우 인상 깊었다. AWS에 회원가입을 하고 서버를 생성하는 것부터 도메인을 구매하고 DNS를 연결하는 과정까지 모든 것이 처음이었다. 이전에는 로컬 환경에서 개발하거나 무료 서비스만 이용해 보았기 때문에, 실제 서버와 도메인을 이용하여 서비스를 공개하는 경험은 프로젝트를 한 단계 완성된 형태로 만들어 주었다. 또한 학습 과정에서만 잠시 사용해 보았던 Docker를 실제 배포 과정에서 활용하고, Nginx와 Gunicorn을 구성하여 서비스를 운영하는 경험도 의미 있는 경험이었다.

이번 프로젝트를 통해 가장 크게 얻은 것은 두 가지이다. 첫 번째는 하나의 프로젝트를 끝까지 완성하고 실제 서비스로 배포해 본 경험이다. 기능 구현에서 끝나는 것이 아니라 운영 가능한 형태로 완성하는 과정을 직접 경험할 수 있었다. 두 번째는 공식 문서를 활용하는 습관이다. 처음에는 AI의 설명을 중심으로 문제를 해결하려 했지만, 프로젝트를 진행하면서 공식 문서를 통해 내용을 검증하고 현재 버전에 맞게 적용하는 과정의 중요성을 자연스럽게 배우게 되었다.

다음 프로젝트에서는 이번 경험을 바탕으로 기획과 설계 단계부터 직접 진행해 보고 싶다. 처음 계획했던 지도(Map) 중심의 프로젝트를 설계 단계부터 실행해보려고 한다. 또한 이번 블로그 프로젝트에서 개발일지에 기록해 둔 개선 사항을 모두 반영하여 기능과 구조를 한 단계 발전시키는 프로젝트를 진행하는 것도 고려하고 있다.
